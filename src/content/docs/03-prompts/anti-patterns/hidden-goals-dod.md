## 症状
- 只说“写个方案/总结/脚本”，却不说“何谓完成”。
- 评审只能靠感觉；模型反复重试仍然不对路。

## 为什么不好
- 模型最擅长“文本合理性”，而非你的隐性指标；没有可验证标准就无法对齐优化目标。

## 改法（工程化）
1) 定义 DoD（可程序化校验）：
   - 结构：必须符合 JSON Schema / Markdown 章节结构；
   - 内容：必须包含 K 个边界用例/引用；禁用/必须包含词；
   - 质量：字数上限、可读性评分、单元测试/脚本校验通过；
2) 在提示开头与结尾各重述一次 DoD；
3) 加“自检步骤”：输出前先列出自检清单并逐项打勾。

## 好例
```md
任务：生成 4 周学习计划（Python 入门）。
输出：必须通过以下 JSON Schema；每周 3 条任务（动宾结构）；总字数 < 250；包含 1 条边界注意事项。
自检：
- [ ] 通过 JSON Schema 校验
- [ ] 每周 tasks=3 且动宾结构
- [ ] 总字数 < 250
- [ ] 边界注意事项已包含
```

## 模板（可复制）
```md
DoD（完成定义）
- 结构：{schema/章节}
- 内容：{必须/禁止}
- 质量：{长度/测试/校验}
自检：
- [ ] 条款1
- [ ] 条款2
- [ ] 条款3
```

## 评估指标
- 结构化校验通过率↑；
- 人工验收时间↓；
- 重试次数↓；
- 一致性：不同批次输出的方差↓。
