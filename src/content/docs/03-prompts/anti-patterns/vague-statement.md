> 记住：模型不是你肚子里的蛔虫。你不说“要它做什么”，它就会用最保守的方式回应，或者自说自话。

## 症状

- 开头只是丢一个模糊现象、想法或心情描述（“我最近工作太累了” / “我们产品转化好像不行”），但不告诉模型要产出什么。
- 既不提供任务类型（分析/生成/归纳/重写/计划），也不限定输出格式或范围。
- 模型只能靠猜测给“安慰、空话或过于泛化”的建议。

## 坏例（不要这样写）

```text
最近我们产品的转化率有点问题。
```

模型常见回应：

- 提一堆“提高转化率的常见方法”大杂烩；
- 追问一些你当下也暂时拿不出的信息；
- 或者直接给出不着边际的“万能建议”。

## 结果

- 输出无法直接落地执行；
- 对后续决策没有帮助；
- 需要多轮来回澄清，效率极低。

## 改法（明确任务 + 输入 + 输出格式 + 约束）

把“你要它做什么”讲清楚。至少说明：

1) 任务：分析/诊断/列方案/改写/规划/生成等；
2) 输入：必要的上下文、数据或样例；
3) 输出格式：列表/表格/JSON/段落；
4) 约束：口吻、长度、是否可假设、信息不足时如何处理。

### 好例 1：诊断问题（分析类）

```text
任务：基于下述漏斗数据，诊断“从到达商品页到加入购物车”这一步骤转化下滑的最可能原因，并列出下一步验证动作。

输入：
- 时间：近 7 天 vs 前 30 天平均
- 指标：到达商品页 UV、加入购物车率、SKU 类别分布、渠道占比
- 备注：我们最近上新了一个高价 SKU，并调整了运费策略

输出格式：
1. 可能原因（按可能性排序，3-5 条）
2. 需要验证的数据/事件（2-4 条）
3. 本周可执行的验证动作（3 条，粒度到“谁做什么”）

约束：
- 如信息不足，请先用 2 条问题澄清再给方案；
- 禁止给与事实无关的空话。
```

### 好例 2：改写内容（生成类）

```text
任务：把下面这段“内部汇报”的口吻，改写成“面向客户邮件”，保持关键信息不变。

输入：
"""
项目进度因为客户侧 API 限流，导致我们本周只完成 60% 里程碑。
"""

输出格式：
- Subject：{英文主题}
- Body：{英文正文，2-3 段}

约束：
- 语气专业、礼貌但不卑微；
- 不承诺具体时间，只承诺“有进展会第一时间同步”。
```

### 模板（可直接复用）

```md
任务：
这次你要完成的任务是：{用动词+对象的一句话描述}

输入：
{贴关键上下文/数据/样例，越少越准}

输出格式：
- {用要点列清楚结构}

约束：
- 语气：{口吻要求}
- 长度：{如“最多 150 字”}
- 信息不足时：{如“先提出 1-2 个澄清问题”}
```

> 原则：哪怕只多写一句“这次你要做的任务是：___”，质量都会显著提升。
